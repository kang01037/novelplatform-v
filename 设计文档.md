双 Token JWT 认证

背景
原有的单 JWT token 认证存在两个问题：
1. 所有 API 端点没有 JWT 过滤拦截，token 签发后实际未做校验
2. 单 token 无法同时满足"短期有效保障安全"和"长期有效提升体验"

设计目标
- 引入 Access Token + Refresh Token 双 token 机制
- 增加 JWT 请求过滤器，拦截需要认证的请求
- Refresh Token 存入 Redis，支持吊销

技术选型
- JWT 库: io.jsonwebtoken (jjwt) 0.11.5
- 存储: Redis（存储 refresh token 哈希，用于吊销校验）
- 签名算法: HMAC-SHA256

Token 设计

Access Token
- 有效期: 30分钟（可通过 jwt.access-expiration 配置）
- 用途: 请求认证，在 Authorization 头中携带
- 载荷: userId, username, tokenType=access

Refresh Token
- 有效期: 7天（可通过 jwt.refresh-expiration 配置）
- 用途: 获取新的 access token
- 载荷: userId, username, tokenType=refresh
- 服务端存储: Redis key = "refresh:token:{md5(token)}"，value = userId

认证流程

1. 登录: 用户名密码验证通过 → 签发 accessToken + refreshToken → refreshToken 哈希存入 Redis → 返回双 token
2. 请求认证: 前端在 Authorization 头携带 Bearer accessToken → JwtAuthenticationFilter 验证 token 有效性及类型
3. Token 刷新: accessToken 过期 → 前端用 refreshToken 调用 /api/auth/refresh → 服务端验证 refreshToken, 检查 Redis 中是否存在 → 轮换签发新双 token → 旧 refreshToken 从 Redis 删除
4. 退出登录: 前端调用 /api/auth/logout → 服务端从 Redis 删除对应的 refreshToken

安全措施
- Token 类型区分: access token 和 refresh token 使用不同 tokenType 声明，防止用 refresh token 冒充 access token
- Refresh Token 轮换: 每次刷新签发新的 refresh token，旧的立即吊销
- 服务端吊销: 刷新和退出时从 Redis 删除，即使 JWT 未过期也无法使用
- 请求过滤: JwtAuthenticationFilter 拦截所有非 GET 和公开路径的请求

实现文件

后端:
- util/JwtUtil.java: 双 token 生成与验证
- config/JwtAuthenticationFilter.java: 请求拦截过滤器
- config/SecurityConfig.java: 安全配置，添加过滤器，定义公开路径
- controller/AuthController.java: 刷新 token 和退出登录接口
- controller/UserController.java: 登录返回双 token
- controller/WechatLoginController.java: 微信登录返回双 token

前端:
- api/index.js: 请求拦截器自动注入 accessToken，响应拦截器自动刷新 token
- Login.vue: 存储 accessToken 和 refreshToken
- App.vue: 退出时调用注销接口，清除双 token
- router/index.js: 路由守卫检查 accessToken
